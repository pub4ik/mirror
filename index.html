Как отправлять баг-репорт

Хороший баг-репорт — это чёткая инструкция, по которой любой человек сможет воспроизвести ту же проблему.

bug report issue template QA

1. Заголовок

Кратко и по делу:

  • Плохо: Помогите!!!
  • Хорошо: [Web] Краш при сохранении профиля

Рекомендуемый формат: [платформа/страница] что ломается + когда

2. Что писать в описании

Скопируйте этот шаблон в новый issue и заполните:

**Окружение:**
- Устройство: (ПК / ноутбук / телефон, модель)
- ОС: (Windows 11, Android 14, iOS, Linux и т.д.)
- Браузер / версия приложения: (Chrome 143, Firefox, Safari…)
- Тип аккаунта (если важно): (гость / пользователь / админ)

**Шаги для воспроизведения:**
1. ...
2. ...
3. ...
4. ...

**Ожидаемый результат:**
Что, по идее, должно происходить.

**Фактический результат:**
Что происходит на самом деле (ошибка, странное поведение и т.д.).

**Повторяемость:**
Каждый раз / иногда / один раз.

**Важность:**
minor  — мешает, но пользоваться можно  
major  — сильно мешает использовать функционал  
blocker — пользоваться приложением невозможно

**Доп. информация:**
Любые наблюдения, которые могут помочь (например:
"если убрать цифры из имени — сохраняется").

**Приложения:**
Скриншоты / видео / логи (если есть).

3. Советы

  • Один баг — один issue.
  • Пишите шаги максимально подробно и по порядку.
  • Если баг редкий, укажите это (иногда / 1 раз из 10).
  • Скриншот или короткое видео часто экономят кучу времени.

Кратко:

Формула хорошего баг-репорта: где случилось → как повторить → что ожидал → что получил.
