Как отправлять правильный баг-репорт

Хороший баг-репорт — это рецепт, по которому любой может воспроизвести ту же проблему, что и ты.

bug report issue template QA

1. Главные принципы

  1. Конкретно — не «ничего не работает», а что именно и при каких условиях.
  2. Повторяемо — чтобы другой человек мог повторить те же шаги.
  3. Понятно — без лишних эмоций, максимум фактов.
  4. Полезно — укажи всё, что реально может помочь: версия, устройство, скрин, лог и т.п.

2. Структура правильного баг-репорта

2.1. Заголовок

Кратко и по делу:

  • Плохо: «Ничего не работает»
  • Хорошо: [Android] Краш приложения при нажатии "Сохранить" в профиле

Формула: [платформа/страница] что сломано + когда

2.2. Окружение (Environment)

Где именно это случилось:

  • Устройство: ПК / телефон / планшет, модель (если знаешь)
  • ОС: Windows 11, Android 14, iOS и т.д.
  • Браузер: Chrome, Firefox, Safari и версия
  • Версия приложения / сайта (если есть номер билда)
  • Аккаунт (если важно): обычный пользователь / админ / гость

Пример:

Окружение:
- Windows 11 x64
- Google Chrome 143.0
- Веб-версия, пользователь без прав администратора

2.3. Шаги для воспроизведения (Steps to reproduce)

Самая важная часть. Нумерованный список действий:

  1. Зашёл на сайт example.com
  2. Нажал «Вход»
  3. Ввёл логин/пароль тестового пользователя
  4. Перешёл в «Профиль»
  5. Изменил имя и нажал «Сохранить»

Чем точнее — тем лучше. Если нужно, укажи тестовые данные (какой текст вводил, какие параметры выбирал и т.п.).

2.4. Ожидаемый результат vs фактический результат

Ожидаемый результат: что, по твоей логике, должно происходить.

Фактический результат: что произошло на самом деле.

Пример:

Ожидаемый результат:
Имя в профиле обновляется, показывается сообщение «Изменения сохранены».

Фактический результат:
Появляется красный экран ошибки, данные не сохраняются.

2.5. Дополнительные детали

Очень полезные вещи:

  • Повторяемость: всегда / иногда / один раз.
  • Серьёзность:
    • Minor — мешает, но жить можно.
    • Major — сильно мешает использовать функционал.
    • Blocker — вообще не даёт пользоваться продуктом.
  • Приложения: скриншоты, короткое видео (запись экрана), логи / консоль, если умеешь это открыть.

3. Готовый шаблон для баг-репорта

Можно использовать как основу в issue-трекере или системе задач.

Заголовок:
[Платформа/страница] кратко в чём проблема

Окружение:
- Устройство:
- ОС:
- Браузер/версия приложения:
- Тип аккаунта (если важно):

Шаги для воспроизведения:
1.
2.
3.
...

Ожидаемый результат:
...

Фактический результат:
...

Повторяемость:
(каждый раз / иногда / один раз)

Важность:
(minor / major / blocker)

Доп. информация:
(заметил, что..., возможно связано с..., если обновить страницу — становится норм и т.п.)

Приложения:
- скриншот/видео/лог (если есть)

4. Пример оформленного баг-репорта

Заголовок:
[Web / Profile] Краш при сохранении изменений в профиле

Окружение:
- Windows 11 x64
- Google Chrome 143.0
- Веб-приложение, пользователь без прав админа

Шаги для воспроизведения:
1. Зайти на https://example.com
2. Нажать "Войти" и авторизоваться под обычным пользователем
3. Перейти в раздел "Профиль"
4. Изменить поле "Имя" на "Test User 123"
5. Нажать кнопку "Сохранить"

Ожидаемый результат:
Имя обновляется, показывается сообщение "Изменения сохранены".

Фактический результат:
Появляется красная ошибка "500 Server Error", данные не сохраняются.

Повторяемость:
Каждый раз (пробовал 5 раз, в разных браузерах).

Важность:
Major (раздел профиля невозможно использовать).

Доп. информация:
Если оставить имя без цифр (например "Test User"), то сохраняется нормально.

Приложения:
- Скриншот ошибки на экране
- Консоль браузера (есть ошибка 500 в ответе /api/profile/update)

5. Чего лучше не делать

  • Не писать «Ничего не работает!!!» без деталей.
  • Не добавлять баг + кучу хотелок в один репорт.
  • Не смешивать несколько разных багов в одном issue.
  • Не оставлять шаги в стиле «я где-то нажал и всё сломалось».
  • Не ограничиваться только эмоциями без фактов.

6. Итог

Формула хорошего баг-репорта: где случилось → как повторить → что ожидал → что получил.
